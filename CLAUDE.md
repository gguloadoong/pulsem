# PulseM — Claude 개발 지침

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 서비스명 | PulseM |
| 한 줄 설명 | 투자의 맥박을 실시간으로 느끼는 게이미피케이션 투자 정보 플랫폼 |
| 문제 정의 | 투자자들이 주식/코인/부동산 정보를 여러 앱에서 파편적으로 확인하며, 중요 규제/소식을 놓치는 문제 |
| 타겟 유저 | 20-40대 투자에 관심 있는 직장인 |
| 핵심 차별점 | 게이미피케이션 + 올인원 투자정보 + 습관 형성 메커니즘 |

## 기술 스택

| 레이어 | 기술 |
|--------|------|
| 프레임워크 | Next.js 15 (App Router) |
| 언어 | TypeScript (strict mode) |
| 스타일링 | Tailwind CSS v4 |
| 애니메이션 | Framer Motion |
| 백엔드/DB | Supabase (Auth, PostgreSQL, Realtime, Storage) |
| 배포 | Vercel |
| 패키지 매니저 | pnpm |

## 코드 작성 규칙

### TypeScript
- `strict: true` 필수. `any` 타입 사용 금지
- 모든 외부 API 응답에 Zod 스키마 검증 적용
- 컴포넌트 Props는 별도 `interface` 정의 (인라인 타입 지양)
- 비동기 함수 반환 타입 명시: `Promise<User | null>`

### 컴포넌트 패턴
- Server Component 우선 원칙 — 필요한 경우에만 `'use client'`
- 데이터 페칭은 Server Component에서 수행
- 클라이언트 인터랙션(애니메이션, 이벤트)만 Client Component로 분리
- 컴포넌트 파일명: PascalCase (`MarketCard.tsx`)
- 훅 파일명: camelCase with `use` prefix (`useMarketData.ts`)

### 스타일링 (Tailwind CSS v4)
- 유틸리티 클래스 직접 사용 — 커스텀 CSS 최소화
- 다크 테마 기본값 (`dark:` prefix 없이 dark 우선 설계)
- Framer Motion 애니메이션은 `variants` 패턴으로 재사용성 확보
- 컴포넌트별 애니메이션 variants를 별도 파일로 분리 (`*.variants.ts`)

### 상태 관리
- 서버 상태: TanStack Query (캐싱, 실시간 업데이트)
- 클라이언트 상태: Zustand (전역 UI 상태, 사용자 설정)
- URL 상태: Next.js `searchParams` (필터, 페이지네이션)
- 폼 상태: React Hook Form + Zod

### API 및 데이터
- 외부 API 호출은 반드시 `/src/lib/api/` 내 클라이언트 함수를 통해서만
- API 응답 캐싱 전략을 주석으로 명시 (`// cache: 5min`)
- 에러는 Error boundary + toast 알림으로 처리
- 실시간 데이터는 Supabase Realtime 또는 WebSocket

### 네이밍 규칙
```
컴포넌트:     PascalCase   → MarketPulseCard
훅:           camelCase    → useStockPrice
유틸 함수:    camelCase    → formatKRW
상수:         SCREAMING_SNAKE_CASE → MAX_WATCHLIST_ITEMS
타입/인터페이스: PascalCase → StockQuote, UserProfile
Supabase 테이블: snake_case → user_achievements
```

### 게이미피케이션 시스템
- 포인트/XP 계산 로직은 `/src/lib/gamification/` 에 집중
- 서버 사이드에서만 포인트 부여 (클라이언트 조작 방지)
- 업적 달성 체크는 Supabase Database Function (트리거) 활용
- 랭킹 업데이트는 Supabase Realtime으로 실시간 반영

## 커밋 규칙

한국어 커밋 메시지, 원자적 커밋 준수.

```
feat: 주식 시세 실시간 업데이트 기능 추가
fix: 코인 가격 소수점 표시 오류 수정
refactor: 시장 데이터 페칭 훅 분리
docs: API 스펙 문서 업데이트
chore: Supabase 마이그레이션 파일 추가
```

## Issue & PR 규칙

`feat:` / `fix:` 작업 시 반드시 Issue 먼저 생성 후 PR 진행.

```
1. gh issue create (라벨: ai-generated + 작업 성격)
2. git checkout -b feature/#이슈번호-설명
3. 작업 완료 후 커밋 & 푸시
4. gh pr create (본문에 "Closes #이슈번호" 포함)
```

PR 본문 마지막:
```
---
🤖 Generated by Claude Code [claude-sonnet-4-6]
```

### 기획 리뷰 (Claude Code 담당)

- 리뷰 종합 시 `.project/PRD.md`를 읽고 PR이 기획과 일치하는지 확인한다
- PRD에 없는 기능이면 "PRD 업데이트 필요" 코멘트를 남긴다
- PRD와 다르게 구현했으면 차이점을 지적하고 의도된 변경인지 확인한다
- CLAUDE.md의 프로젝트 개요/타겟 유저/핵심 가치 기준으로 서비스 방향성을 판단한다
- `feat:` PR에서 스펙이 변경됐으면 같은 PR에서 PRD도 함께 수정하도록 권장한다

### PR 리뷰 대응 (절대 건너뛰지 마라)

**⚠️ PR 생성 후 즉시 머지 금지. 반드시 봇 리뷰를 확인하고 대응한 뒤 머지하라.**

```
PR 생성 후 필수 절차:
1. PR 생성 → 최소 2분 대기 (봇 리뷰 도착 시간)
2. gh api 로 봇 리뷰 코멘트 확인 (Gemini, CodeRabbit, Copilot)
3. "리뷰 종합" 코멘트 작성 — 봇 간 동의/충돌/채택/기각 정리
4. @coderabbitai 멘션으로 토론 질문을 던진다
5. 최소 2분 대기 → 봇 응답 확인 (gh api 로 새 코멘트 체크)
6. 봇 응답을 읽고 "최종 판단" 코멘트 작성 후 머지

토론 종료 규칙 (무한 루프 방지):
- 토론은 최대 1회전: 종합 코멘트(1회) → 봇 응답 확인(1회) → 최종 판단 → 머지
- 봇 응답에 새로운 질문을 던지지 않는다 — 읽고 판단만 한다
- 보안/버그 지적 → 코드 수정 후 머지 (추가 토론 없이)
- 스타일/취향 제안 → 채택 or 기각 근거 한 줄 남기고 머지
- 봇 응답이 2분 내 안 오면 기존 리뷰만으로 판단하고 머지
```
