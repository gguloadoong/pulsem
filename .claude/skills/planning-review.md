# 기획 리뷰 스킬

## 목적
PRD(제품 요구사항 문서)와 기획 문서를 팀이 함께 검토하여 개발 착수 전에 요구사항 품질, 실현 가능성, 완결성을 확인한다.

## 트리거 조건
- 신규 기능 PRD 초안 완성 시
- 기존 기능 대규모 변경 기획 완성 시
- 킥오프 미팅 전 PRD 사전 검토 시
- 스프린트 플래닝 전 백로그 이슈 품질 검토 시

---

## 실행 순서

### Step 1: PM의 PRD 자체 검토 (리뷰 요청 전)

**완성도 체크리스트:**
- [ ] 배경/목적: 왜 이 기능을 만드는가?
- [ ] 목표 KPI: 성공을 어떻게 측정하는가?
- [ ] 사용자 스토리: 누가, 무엇을, 왜 하려는가?
- [ ] 수용 기준: 기능이 완료된 조건이 구체적인가?
- [ ] 비기능 요건: 성능, 보안, 접근성 요건이 있는가?
- [ ] 스코프 아웃: 이번 범위에서 제외된 것이 명시되어 있는가?
- [ ] 의존성: 다른 기능/시스템과의 의존관계가 기술되어 있는가?
- [ ] 와이어프레임 링크: Figma 링크 또는 스케치 첨부
- [ ] 우선순위: RICE 스코어 또는 근거 기반 우선순위

### Step 2: 팀 리뷰 세션 (최대 60분)

**참석자:**
- 필수: PM, FE, BE
- 선택: Designer, QA, Strategist (관련도 높을 시)

**리뷰 순서:**

**0-10분: PM 설명**
- PRD 핵심 내용 구두 요약 (문서 읽기가 아닌 설명)
- 특별히 팀 의견이 필요한 포인트 미리 제시

**10-30분: 질문 & 검토**
각 팀원이 담당 관점에서 검토:

*FE 정민우 검토 포인트:*
- 기술적 실현 가능성
- 추정 스토리포인트
- 성능 요건 달성 가능성
- 외부 라이브러리/API 의존성

*BE 최영진 검토 포인트:*
- 데이터 모델 설계 가능성
- API 설계 복잡도
- 보안/인증 요건
- 실시간 데이터 요건

*Designer 이수진 검토 포인트:*
- 사용자 경험 흐름 자연스러움
- 디자인 시스템 내 구현 가능성
- 접근성 요건

*QA 한소희 검토 포인트:*
- 수용 기준의 테스트 가능성
- 누락된 엣지케이스
- 테스트 환경 요건

**30-50분: 이슈 해결**
- 모호한 수용 기준 명확화
- 기술적 제약으로 인한 스코프 조정
- 우선순위 조정 논의

**50-60분: 합의 및 다음 단계**
- PRD 수정 사항 목록 확정
- 수정 완료 후 재리뷰 필요 여부
- 킥오프/개발 착수 일정 확정

### Step 3: PRD 수정 및 최종 승인
- PM이 리뷰 피드백 반영하여 PRD 업데이트
- 주요 변경 시 재리뷰
- 최종 PRD에 팀 승인 표시 (Notion 승인 버튼 또는 Slack 확인 이모지)

## PRD 품질 기준

### 수용 기준(Acceptance Criteria) 작성 기준
좋은 수용 기준의 조건:
- **Given-When-Then 형식** 사용 권장
  ```
  Given [전제 조건],
  When [행동],
  Then [기대 결과].
  ```
- 테스트 가능한 구체적 상태 기술
- 숫자로 측정 가능한 조건 포함

나쁜 수용 기준 예시 (이런 거 쓰면 박서준한테 혼남):
```
❌ "빠르게 로딩된다"
❌ "사용자가 편리하게 사용한다"
❌ "정상적으로 동작한다"
```

좋은 수용 기준 예시:
```
✅ "종목 카드 클릭 시 상세 페이지가 300ms 이내에 로딩된다"
✅ "잘못된 이메일 형식 입력 시 '이메일 형식이 올바르지 않습니다' 에러 메시지가 입력 필드 아래 표시된다"
✅ "퀘스트 완료 시 XP가 즉시 증가하고 화면 상단에 '+50 XP' 애니메이션이 표시된다"
```

## 출력 형식

```markdown
# 기획 리뷰: [기능명]
**리뷰 일시:** YYYY-MM-DD
**PRD 링크:** [Notion 링크]
**리뷰어:** [참석자 목록]

## PRD 완성도 평가
- 배경/목적: ✅ 명확 / ⚠️ 보완 필요 / ❌ 누락
- 성공 지표: ✅ / ⚠️ / ❌
- 수용 기준: ✅ / ⚠️ / ❌
- 비기능 요건: ✅ / ⚠️ / ❌
- 와이어프레임: ✅ / ⚠️ / ❌

## 필수 수정 사항
- [ ] [수정 내용] — 담당: 박서준 — 기한: [날짜]

## 기술 검토 결과
**FE (정민우):** [스토리포인트 추정 + 기술 리스크]
**BE (최영진):** [데이터 모델 이슈 + API 복잡도]
**디자이너 (이수진):** [UX 이슈 + 디자인 방향]
**QA (한소희):** [누락 엣지케이스 + 테스트 가능성]

## 합의된 스코프 변경
- [변경 내용]: [이유]

## 개발 착수 가능 여부
**YES ✅ / NO (수정 후 가능) ⚠️**
```

---

## 규칙
- PRD 없는 기능의 스프린트 배정 금지
- 수용 기준이 없거나 모호한 PRD는 리뷰에서 반려
- 기술 검토 없이 스토리포인트 확정 금지
- PRD 수정 후 주요 변경 시 재리뷰 필수
