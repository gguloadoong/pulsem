# 디자인 리뷰 스킬

## 목적
디자인이 PRD 수용 기준을 충족하는지, 사용성·접근성·브랜드 일관성을 갖추었는지 확인하고 개발 전달 준비 상태를 검증한다.

## 트리거 조건
- 신규 화면 또는 컴포넌트 디자인 완료 시
- 기존 화면 대규모 리디자인 완료 시
- 개발 핸드오프 전 최종 확인
- 디자인 QA (개발 구현 후 디자인 비교)

---

## 실행 순서

### Step 1: 리뷰 준비 (Designer, 리뷰 전)
- Figma 링크 + 프레임 목록 Slack 공유
- PRD 링크 함께 첨부
- 리뷰 범위 명시 (신규 컴포넌트 X개, 수정 화면 Y개)
- 리뷰어: PM (수용 기준), FE (구현 가능성), QA (테스트 케이스 기반)

### Step 2: 자체 체크리스트 (Designer 먼저 실행)

**기능 완성도:**
- [ ] PRD의 모든 수용 기준에 대응하는 화면 존재
- [ ] 성공 상태, 로딩 상태, 에러 상태, 빈 상태 모두 디자인됨
- [ ] 모바일(360px), 태블릿(768px) 반응형 확인

**디자인 시스템 일관성:**
- [ ] 색상 토큰 사용 (하드코딩 없음)
- [ ] 타이포그래피 스케일 준수
- [ ] 컴포넌트 라이브러리에서 가져온 컴포넌트 사용
- [ ] 간격 시스템(4px 배수) 준수

**접근성:**
- [ ] 텍스트-배경 색상 대비 4.5:1 이상 (일반 텍스트)
- [ ] 대형 텍스트(18px+) 대비 3:1 이상
- [ ] 터치 타겟 최소 44×44px
- [ ] 포커스 인디케이터 존재 (키보드 네비게이션)
- [ ] 아이콘 전용 버튼에 레이블 or aria-label 명시

**인터랙션 명세:**
- [ ] 호버/포커스/액티브/비활성 상태 디자인
- [ ] 애니메이션 있으면 ProtoPie 또는 Figma 프로토타입 연결

### Step 3: 팀 리뷰 세션 (최대 45분)

**0-5분: 컨텍스트 공유 (Designer)**
- 이 화면이 해결하는 사용자 문제
- 디자인 결정의 핵심 근거
- 특별히 검토 요청하는 포인트

**5-25분: 화면별 리뷰**
- PM: PRD 수용 기준 대조 확인
- FE: 구현 가능성, 애니메이션 퍼포먼스 우려
- QA: 엣지케이스 화면 존재 여부

**25-40분: 피드백 통합**
- 피드백을 3가지로 분류:
  - **필수 수정**: 배포 전 반드시 반영
  - **권장 수정**: 이번 스프린트 내 반영 권장
  - **추후 개선**: 백로그 등록

**40-45분: 다음 단계 확정**
- 수정 완료 예상 일자
- 재리뷰 필요 여부
- 개발 핸드오프 일자

### Step 4: 디자인 QA (개발 구현 후)
- 실제 구현 화면과 Figma 픽셀 비교
- 반응형 뷰포트별 확인
- 다크모드/라이트모드 전환 확인
- 실제 데이터(긴 텍스트, 빈 데이터) 확인

## 출력 형식

```markdown
# 디자인 리뷰: [화면/기능명]
**일시:** YYYY-MM-DD
**Figma:** [링크]
**리뷰어:** 이수진(디자이너), 박서준(PM), 정민우(FE), 한소희(QA)

## 리뷰 결과

### 필수 수정 (배포 전)
- [ ] [화면명] [문제 설명] → [수정 방향]
  - 예: 종목 카드 에러 상태 미디자인 → 에러 아이콘 + 재시도 버튼 추가

### 권장 수정 (이번 스프린트)
- [ ] [내용]

### 추후 개선 (백로그)
- [ ] [내용]

## 접근성 체크 결과
- 색상 대비: 통과 ✅ / 실패 ❌ [상세]
- 터치 타겟: 통과 ✅ / 실패 ❌ [상세]
- 포커스: 통과 ✅ / 미확인 ⏳

## 핸드오프 준비 상태
- 개발 핸드오프 가능: YES / NO (수정 후 가능)
- 수정 완료 예정: YYYY-MM-DD
- 재리뷰 필요: YES / NO
```

---

## 규칙
- 디자인 리뷰 없이 개발 착수 금지
- 접근성 기준 미달 시 필수 수정으로 분류 (권장 아님)
- 픽셀장인이 수용 기준 기반 화면 완성 전 리뷰 요청 금지
- 리뷰 코멘트는 Figma 코멘트 + 리뷰 문서 양쪽에 기록
